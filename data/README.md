# Data Directory

This directory contains all data files used and generated by the EpiAgent Skills workflows.

## Directory Structure

```
data/
├── raw/              # Raw, unprocessed data
│   ├── simulated_linelist.qs
│   └── simulated_contacts.qs
└── processed/        # Cleaned and validated data
    └── cleaned_tagged_linelist.qs
```

## Data Sources

### Raw Data (`raw/`)

Raw data files are the initial inputs to analysis pipelines. In the example workflow, these are **simulated** outbreak data generated using the `simulist` package.

#### `simulated_linelist.qs`
- **Format**: R object serialized with `qs` package
- **Type**: Data frame / tibble
- **Source**: Generated by `simulist::sim_outbreak()`
- **Description**: Simulated outbreak line list containing individual case records

**Columns**:
- `id`: Unique case identifier (character)
- `date_onset`: Date of symptom onset (Date)
- `date_reporting`: Date case was reported (Date)
- `date_admission`: Date admitted to hospital (Date, may be NA)
- `date_outcome`: Date of outcome (recovery/death) (Date, may be NA)
- `outcome`: Case outcome - "recovered" or "died" (character)
- `sex`: Biological sex - "m" or "f" (character)
- `age`: Age in years (numeric)
- `date_infection`: Date of infection (Date, only in simulations)
- `date_first_contact`: Date of first contact (Date)

**Size**: Typically 100-500 rows depending on simulation parameters

#### `simulated_contacts.qs`
- **Format**: R object serialized with `qs` package
- **Type**: Data frame / tibble
- **Source**: Generated by `simulist::sim_outbreak()`
- **Description**: Contact tracing data showing transmission chains

**Columns**:
- `from`: ID of infector (character)
- `to`: ID of infectee (character)
- `age_from`: Age of infector (numeric)
- `age_to`: Age of infectee (numeric)
- `date_first_contact`: Date of transmission event (Date)
- `was_case`: Whether contact became a case (logical)
- `generation`: Transmission generation number (integer)

**Note**: Number of contacts typically exceeds number of cases.

### Processed Data (`processed/`)

Processed data has been cleaned, validated, and standardized for analysis.

#### `cleaned_tagged_linelist.qs`
- **Format**: R object serialized with `qs` package
- **Type**: linelist object (from `linelist` package)
- **Source**: Output from `cleanepi::clean_data()` and `linelist::make_linelist()`
- **Description**: Cleaned and tagged line list ready for epidemiological analysis

**Transformations Applied**:
1. Column name standardization (snake_case)
2. Missing value replacement
3. Data validation checks
4. Semantic tagging of epidemiological variables

**Linelist Tags** (metadata):
- `id`: Case identifier
- `date_onset`: Symptom onset date
- `date_reporting`: Reporting date
- `date_admission`: Hospital admission date
- `date_outcome`: Outcome date
- `outcome`: Survival status
- `gender`: Biological sex (mapped from 'sex')
- `age`: Age in years

**Validation**: All tagged fields validated with `linelist::validate_linelist()`

## File Formats

### `.qs` Format

All data files use the `.qs` format (Quick Serialization):

**Advantages**:
- Fast read/write (faster than RDS)
- Excellent compression
- Preserves R object structures perfectly
- Cross-platform compatible

**Usage**:
```r
# Read
library(qs)
linelist <- qread(here::here("data", "raw", "simulated_linelist.qs"))

# Write
qsave(linelist, here::here("data", "raw", "simulated_linelist.qs"))
```

**Alternative Formats**:
```r
# Export to CSV for external tools
readr::write_csv(linelist, "linelist.csv")

# Export to RDS for broader R compatibility
saveRDS(linelist, "linelist.rds")
```

## Data Quality

### Raw Data Quality Checks
- All required date columns present
- ID uniqueness verified
- Date consistency (onset ≤ reporting ≤ outcome)
- Age values reasonable (0-120)
- Outcome values valid ("recovered" or "died")

### Processed Data Quality Checks
- No duplicate IDs
- Standardized column names
- Consistent date formats (ISO 8601)
- Tagged fields validated
- Missing data patterns documented

## Data Privacy

**Important**: These are simulated data files for demonstration purposes only.

If working with real outbreak data:
- **Never commit real patient data** to version control
- Ensure `.gitignore` excludes all data files
- Use de-identified data only
- Follow institutional data protection policies
- Consider encryption for sensitive data

## Reproducibility

### Regenerating Simulated Data

To regenerate the example data with the same parameters:

```r
# Run the complete workflow
source("ebola_workflow.R")

# Or run simulation only
source("ebola_workflow.R")
# Data will be saved to data/raw/
```

**Random Seed**: The workflow uses `set.seed(42)` for reproducibility. Using the same seed will generate identical data.

### Custom Simulations

To generate data with different parameters:

```r
library(simulist)
library(epiparameter)

# Define custom parameters
contact_dist <- epiparameter(
  disease = "Your Disease",
  epi_name = "contact distribution",
  prob_distribution = create_prob_distribution(
    prob_distribution = "pois",
    prob_distribution_params = c(mean = 3)  # Different R0
  )
)

# Simulate with custom parameters
outbreak <- sim_outbreak(
  contact_distribution = contact_dist,
  outbreak_size = c(50, 200),  # Smaller outbreak
  outbreak_start_date = as.Date("2024-06-01")
)

# Save
qs::qsave(outbreak$linelist, here::here("data", "raw", "custom_linelist.qs"))
```

## External Data Sources

To use this workflow with real outbreak data:

1. **Format your data** with these required columns:
   - `id` - Case identifier
   - `date_onset` - Symptom onset date (YYYY-MM-DD)
   - `outcome` - "recovered" or "died"

2. **Save in compatible format**:
   ```r
   qs::qsave(your_data, here::here("data", "raw", "real_outbreak.qs"))
   ```

3. **Modify workflow** to read your data instead of simulating.

## Size and Performance

**Typical File Sizes**:
- `simulated_linelist.qs`: 20-100 KB (100-500 cases)
- `simulated_contacts.qs`: 50-200 KB (depends on transmission intensity)
- `cleaned_tagged_linelist.qs`: Similar to raw (compression efficient)

**Performance**:
- Read time: <1 second for typical outbreak sizes
- Write time: <1 second
- Memory usage: ~5-10 MB for 500 cases

For larger outbreaks (>10,000 cases), consider:
- Chunked processing
- Database storage (e.g., DuckDB, SQLite)
- Parquet format for columnar access
